Aquisitions API

A Node.js REST API for user registration and authentication, built with Express, Drizzle ORM, and PostgreSQL on Neon.

Runtime Node.js (ESM)
Framework Express v5
Database PostgreSQL / Neon
Pattern Layered MVC
1

Big Picture

This is a REST API backend built for handling user acquisition flows — primarily user registration and authentication. It exposes HTTP endpoints consumed by a frontend client (web or mobile) and connects to a serverless PostgreSQL database hosted on Neon.

What type of project?

A Node.js REST API. No server-side rendering, no HTML — pure JSON over HTTP. Designed to be consumed by a separate frontend client.

What problem does it solve?

Manages the user acquisition lifecycle: registration with validation, secure password storage, JWT issuance, and (planned) sign-in/sign-out flows.

Current state

Early-stage. Sign-up is fully implemented. Sign-in and Sign-out endpoints exist as stubs and are not yet functional.

Deployment target

Serverless-friendly. Uses Neon's HTTP driver (@neondatabase/serverless), which is optimized for ephemeral/serverless environments.

Core Architecture

The project follows a Layered Architecture (also called N-Tier). Each layer has a single responsibility and communicates only with the layer directly below it. This is a classic pattern for REST APIs that keeps concerns separated and easy to test.

🌐 Client
Browser / Mobile / Postman
External
↓ HTTP Request
🛣️ Routes Layer
src/routes/auth.routes.js
URL → Handler
↓ calls controller
🎮 Controller Layer
src/controllers/auth.controller.js
Validate + Orchestrate
↓ calls service
⚙️ Service Layer
src/services/auth.service.js
Business Logic
↓ queries DB
🗄️ Data Layer
src/models/users.model.js + src/config/database.js
Drizzle ORM + Neon
↑ Result bubbles back up

Why this pattern? Each layer only knows about the one below it. Controllers don't touch the database directly — they call services. Services don't know about HTTP — they only speak domain logic. This makes each piece independently testable and replaceable.

Key Components

app.js Express Application Bootstrap

The central Express setup file. Applies all middleware in order and mounts route groups. It does not start the server — that is server.js's job, keeping concerns separated.

helmet cors express.json cookieParser morgan→winston /api/auth router
auth.routes.js Route Definitions

Pure URL-to-handler mapping. No logic lives here. Uses express.Router() mounted at /api/auth. Currently defines three routes — only sign-up is wired to a real controller.

POST /api/auth/sign-up   → signup controller  ✅ Implemented
POST /api/auth/sign-in   → inline stub         🚧 Placeholder
POST /api/auth/sign-out  → inline stub         🚧 Placeholder
auth.controller.js Request Orchestration

The controller is the "traffic cop". It handles the HTTP concerns: reads the request body, runs Zod validation, delegates to the service, then constructs and sends the HTTP response. It never writes SQL or business rules.

validates with Zod calls createUser() signs JWT sets cookie returns 201 JSON
auth.service.js Business Logic & DB Access

The only layer that talks to the database. Contains hashPassword() (bcrypt, 12 rounds) and createUser(). Checks for duplicate emails, hashes the password, inserts the row, and returns the new user — with no HTTP knowledge.

duplicate-email check bcrypt hash Drizzle INSERT returns user (no password)
auth.validations.js Input Schemas (Zod)

Defines the shape and rules for incoming request bodies using Zod schemas. signupSchema enforces valid email (auto-lowercased), max lengths, and role enum. signinSchema is ready for when sign-in is implemented.

signupSchema: { name, email, password, role: 'user' | 'admin' }
signinSchema: { email, password }
jwt.js

Wraps jsonwebtoken into a clean object with .sign(payload) and .verify(token). Reads JWT_SECRET and JWT_EXPIRES_IN from env. Falls back to a development-only default secret.

cookies.js

Cookie utility object with .set(), .get(), .clear(). Enforces httpOnly, secure (production only), sameSite: strict, and 15-minute expiry on all cookies.

format.js

Converts Zod's nested error structure (ZodError.issues[]) into a flat, comma-separated human-readable string for API error responses.

logger.js

Winston logger writing error.log and combined.log to disk. In non-production environments adds a colorized console transport. Morgan HTTP logs are piped through Winston.

Error propagation: Service errors are thrown and caught by the controller's try/catch. Known errors (e.g., "email already exists") return a 409. Unknown errors are forwarded to Express via next(e) for global error handling.

Execution Flow

A step-by-step walkthrough of the most complete flow in the system: POST /api/auth/sign-up.

  • 1
    Process boots
    src/index.js loads dotenv, which reads .env and injects PORT, DATABASE_URL, NODE_ENV into process.env. Then imports server.js.
    node src/index.js → Listening on http://localhost:3000
  • 2
    Request arrives
    A client sends POST /api/auth/sign-up with a JSON body. Express processes it through the middleware stack: Helmet sets security headers → CORS adds origin headers → express.json() parses the body → Morgan logs the request via Winston.
  • 3
    Router matches
    app.use('/api/auth', authRouter) sends the request to auth.routes.js, which matches POST /sign-up and calls the signup controller function.
  • 4
    Validation
    Controller calls signupSchema.safeParse(req.body). Zod checks types, formats (valid email), lengths (max 255), and normalizes (email to lowercase, trim whitespace). If it fails, returns 400 immediately with a formatted error message.
    ❌ { "error": "Validation Failed", "details": "Invalid email" } → 400
  • 5
    Service: duplicate email check
    createUser() runs db.select().from(users).where(eq(users.email, email)).limit(1). If a user already exists, throws an error that the controller catches and returns as 409.
    ❌ { "error": "Email already exist" } → 409
  • 6
    Password hashing
    hashPassword() calls bcrypt.hash(password, 12). Cost factor 12 means the hash takes ~300ms — slow enough to resist brute-force, fast enough for a web API.
  • 7
    Database insert
    Drizzle runs INSERT INTO users (name, email, password, role) VALUES (...) RETURNING (id, name, email, role, created_at) over Neon's HTTP transport. Returns the new user row (password hash excluded from the return projection).
  • 8
    JWT issued
    Controller calls jwttoken.sign({ id, email, role }). Payload is signed with JWT_SECRET and expires in JWT_EXPIRES_IN (default 1 day).
  • 9
    Cookie set & response
    Token is stored in an httpOnly, sameSite: strict, secure (in production) cookie with a 15-minute expiry. The JSON response excludes sensitive data.
    ✅ { "message": "User Registered", "user": { id, name, email, role } } → 201

Strengths & Tradeoffs

✅ Strengths
  • ✓ Clear separation of concerns. Each file has one job — routes, controllers, services, and models never bleed into each other.
  • ✓ Solid security defaults. Helmet headers, httpOnly cookies, bcrypt at cost 12, Zod input sanitization, and CORS — all set up out of the box.
  • ✓ Serverless-ready DB. Neon's HTTP driver means no persistent connection pool — perfect for serverless/edge deployments where cold starts are frequent.
  • ✓ Clean import system. Subpath imports (#config/*) eliminate fragile ../../ paths and make refactoring safe.
  • ✓ Structured logging from day one. Winston with file transports and Morgan piped through it means logs are queryable and persistent.
  • ✓ Code quality tooling. ESLint + Prettier integrated means a consistent codebase as the team grows.
⚠️ Watch Out For
  • ⚠ No global error handler. next(e) is called in controllers, but there's no app.use((err, req, res, next) => {...}) to catch it — unhandled errors crash Express.
  • ⚠ Sign-in/Sign-out are stubs. The two remaining auth flows are not implemented — the API is not functional end-to-end yet.
  • ⚠ JWT secret fallback. jwt.js falls back to a hardcoded secret string if JWT_SECRET is not set. This must be set before any production deployment.
  • ⚠ No rate limiting. The sign-up endpoint has no rate limiting — it's open to abuse/spam registrations without something like express-rate-limit.
  • ⚠ 15-minute cookie vs 1-day JWT. The cookie expires in 15 min but the JWT itself is valid for 1 day — a mismatch that could cause confusing auth behavior.
  • ⚠ No tests. Zero test files exist. The ESLint config includes a tests/** glob rule, suggesting tests are planned but not yet written.

Final Summary

The 2-sentence pitch

This is an early-stage Node.js REST API that handles user registration using a clean layered architecture — requests flow from Express routes → controllers (validation) → services (business logic) → Drizzle ORM → Neon PostgreSQL, with JWTs issued on sign-up and stored in secure HTTP-only cookies. Sign-up is the only fully implemented feature; sign-in and sign-out are placeholders, and the next priorities should be completing those flows, adding a global error handler, rate limiting, and a test suite.

Scripts quick reference: npm run dev — start with hot-reload · npm run lint — check code style · npm run db:studio — open Drizzle Studio · npm run db:generate — generate migrations · npm run db:migrate — apply migrations

Before going to production: Set JWT_SECRET to a long random string in your environment, ensure NODE_ENV=production (activates secure cookies), add a global Express error handler, implement rate limiting on auth routes, and add the JWT_EXPIRES_IN env var.
